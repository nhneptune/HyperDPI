# HyperDPI

High-Performance Deep Packet Inspection (DPI) gateway using DPDK + Hyperscan.

HyperDPI simulates how a Firewall / IDS / IPS / 5G UPF inspects traffic in-line: it reads packets from a `traffic.pcap` file through DPDK's `net_pcap` virtual device, classifies each packet through L2→L7 parsing, extracts the HTTP URL or TLS SNI hostname, matches it against a rule set with Intel Hyperscan, and forwards or drops accordingly — all on a pinned-core, lock-free pipeline built for line-rate throughput.

Full assignment spec: [`hyperdpi_description.txt`](hyperdpi_description.txt) (Vietnamese, authoritative). Implementation/test history and validated KPI results: [`version_01.md`](version_01.md) (Vietnamese).

## Architecture

```
traffic.pcap --(net_pcap, infinite_rx=1)--> RX Thread --(hash 5-tuple)--> N x rte_ring --> Worker Thread(s)
                                                                                              |
                                                                     parse L2-L4 -> reassembly/L7 extract -> hs_scan
                                                                                              |
                                                                          FORWARD -> TX ring -> TX Thread -> rte_eth_tx_burst
                                                                          DROP    -> rte_pktmbuf_free
```

- **RX Thread** — pure polling (`rte_eth_rx_burst`, burst=32), hashes each packet's 5-tuple and looks it up in a load-aware bucket→worker table to pick a worker ring (keeps every segment of a TCP flow on the same worker).
- **Worker Threads** (2–4, configurable) — each owns an independent `rte_ring` and a per-flow TCP/TLS reassembly table, does L2–L7 parsing, and runs the Hyperscan scan.
- **TX Thread** — gathers up to a full burst before a single `rte_eth_tx_burst` call.
- **Stats Thread** — prints aggregated per-core counters (throughput, PPS, classification, forward/drop, per-worker PPS, bucket distribution) every 1s, and periodically rebalances idle hash buckets away from the most-loaded worker (RSS++-inspired load-aware steering — see "Load-aware RX steering" below).

See [`.claude/CLAUDE.md`](.claude/CLAUDE.md) for the full design writeup (flow reassembly, fail-open policy, Hyperscan scratch handling, load-aware bucket rebalancing, etc).

## Requirements

- Linux, DPDK (tested with 23.11) and Intel Hyperscan dev packages resolvable via `pkg-config libdpdk libhs`
- `gcc`/`make`
- `python3` + `scapy` (only needed to regenerate `pcap/traffic.pcap`)
- root access (DPDK hugepages + device access)

## Step-by-step: build

```sh
cd HyperDPI
make clean && make
```
No output from `make` beyond the compile/link commands means a clean build. Produces the `./hyperdpi` binary.

## Step-by-step: hugepages

DPDK needs hugepage-backed memory. Run once per boot:

```sh
sudo bash -c "echo 1 > /proc/sys/vm/compact_memory"
sudo bash -c "echo 512 > /sys/devices/system/node/node0/hugepages/hugepages-2048kB/nr_hugepages"
cat /sys/devices/system/node/node0/hugepages/hugepages-2048kB/nr_hugepages   # confirm it reads 512
```
The `compact_memory` step matters: on a fragmented host (seen in practice on a WSL2 dev VM), allocating hugepages directly can silently return fewer pages than requested, which then makes `rte_eal_init()` fail with "Cannot get hugepage information." Always verify the readback before proceeding.

## Step-by-step: generate test data

Already checked in, but to regenerate:
```sh
python3 scripts/gen_pcap.py
```
Produces `pcap/traffic.pcap` with 4 flows: HTTP DROP, HTTPS/SNI DROP (single TCP segment), FORWARD (no rule match), and HTTPS/SNI DROP split across **two** TCP segments (exercises the reassembly path).

`config/patterns.txt` ships with the spec's example rule set:
```
HTTP,http://malware.com.*,DROP
HTTP,http://trusted-partner.com.*,FORWARD
HTTPS,HOST://bad-gambling.net.*,DROP
HTTPS,HOST://.*facebook.com.*,FORWARD
```

## Step-by-step: run the 3 required test scenarios

```sh
sudo stdbuf -oL -eL ./hyperdpi -l 0-6 -n 4 \
  --vdev=net_pcap0,rx_pcap=pcap/traffic.pcap,tx_pcap=pcap/out.pcap \
  -- -p config/patterns.txt -w 2 2>&1 | tee logs/functional_test.log
```
Watch for lines like:
```
[Worker 1] MATCH    http://malware.com/evil -> DROP
[Worker 1] MATCH    HOST://bad-gambling.net -> DROP
[Worker 1] NO MATCH http://example.com/ -> FORWARD
```
Once you've seen the `MATCH`/`NO MATCH` lines and at least one `Throughput:` stats block, press `Ctrl+C` to stop (the process idles after the pcap is exhausted since `infinite_rx` isn't set here).

- `-l 0-6`: lcore 0 = main, 1-6 = 6 lcores for RX+Worker+TX+Stats. Minimum needed is `1 + num_workers + 1 + 1`; adjust the range if you change `-w`.
- `-w`: number of worker threads, 2-4.
- `-p`: path to the rules file.
- `-B`: disable load-aware bucket rebalancing (steering still goes through the bucket table, it just never rebalances) — see "Load-aware RX steering" below.
- `-M <nb_mbufs>`: override the mbuf pool size — only needed for `infinite_rx=1` against large fixtures (see below); the default is fine for `traffic.pcap`.

## Step-by-step: 30-minute soak test

Tests the KPI throughput/PPS targets and memory stability (spec section 7). **Do not pass `tx_pcap` here** — combined with `infinite_rx=1` it grows unbounded (observed 8.8GB in 60s in testing).

Terminal 1 — launch:
```sh
mkdir -p logs
TS=$(date +%Y%m%d_%H%M%S)
sudo stdbuf -oL -eL ./hyperdpi -l 0-6 -n 4 \
  --vdev=net_pcap0,rx_pcap=pcap/traffic.pcap,infinite_rx=1 \
  -- -p config/patterns.txt -w 2 > logs/soak_$TS.log 2>&1 &
echo $! > /tmp/hyperdpi_sudo.pid
```

Terminal 2 — sample RSS every 30s for 30 minutes (60 samples). Use `pgrep -x hyperdpi` for the PID, **not** a pattern match on the launch command — the latter also matches the wrapping `sudo`/`stdbuf` processes and silently samples the wrong process:
```sh
PID=$(pgrep -x hyperdpi)
RSS_LOG=logs/soak_rss_$TS.log
for i in $(seq 1 60); do
  sleep 30
  echo "$(date +%T) RSS=$(ps -o rss= -p $PID) KB" >> "$RSS_LOG"
done
```

After 30 minutes, stop it:
```sh
sudo kill -TERM $(pgrep -x hyperdpi)
sleep 2
tail logs/soak_$TS.log   # should show "exiting on lcore" for RX/Worker/TX/Stats, no errors
```

## Step-by-step: generate the KPI report

```sh
python3 scripts/kpi_report.py logs/soak_$TS.log logs/soak_rss_$TS.log
```
Prints min/avg/max Throughput and PPS against the spec's targets (400 Mbps–1.0 Gbps, 100K–500K PPS), a PASS/FAIL verdict for each, final packet counters, whether a clean shutdown was observed, and (if an RSS log was passed) first/last RSS with a memory-stability verdict. Designed to be screenshotted directly for a report.

Validated results from a real 30-minute run are recorded in [`version_01.md`](version_01.md).

## Load-aware RX steering (RSS++-inspired)

Plain `hash % num_workers` has no feedback loop — an elephant flow (many packets, one 5-tuple) parked on one worker stays there regardless of load, even while other workers idle. RX instead looks the hash up in a 256-entry bucket→worker table; the Stats thread rebalances buckets that have gone idle (no traffic in the last 200ms) away from the busiest worker toward the least-busy one, every 1s tick. **A bucket a live flow is actively using is never moved** — this Phase 1 only redirects idle buckets, it doesn't migrate an in-flight flow's reassembly state. Full design writeup, the underflow bug that was caught and fixed during review, and the full "before"/"after" measurements: [`.claude/CLAUDE.md`](.claude/CLAUDE.md) and [`version_01.md`](version_01.md) §6.

Check `nproc` before picking core args below — on a low-CPU-count box (e.g. 4 CPUs), `-l 0-6` won't work (EAL needs that many distinct real CPU IDs). Use `--lcores '(0-6)@(0-3)'` instead, which maps the same 7 logical lcores onto whatever physical CPUs exist, letting the OS time-share them — confirmed functionally identical, just slower.

### Verifying the imbalance problem ("before") and the rebalance fix ("after")

Fixture size is a real tradeoff: it needs enough packets that one `infinite_rx` loop pass takes noticeably longer than the 200ms idle threshold (this app can replay 10M+ pps even from a file), but `infinite_rx=1` requires the mbuf pool (`-M`) to hold the *entire* file's packet count in memory at once. Pick a size your RAM can back with hugepages — e.g. ~3.8M packets needs roughly 8-9GB of hugepages:

```sh
python3 scripts/gen_elephant_pcap.py --out pcap/elephant.pcap --mice 10 --elephant-gap 380000
sudo bash -c "echo 4608 > /sys/devices/system/node/node0/hugepages/hugepages-2048kB/nr_hugepages"   # ~9GB
```

**Before** (rebalancing disabled):
```sh
sudo stdbuf -oL -eL ./hyperdpi --lcores '(0-6)@(0-3)' -n 4 \
  --vdev=net_pcap0,rx_pcap=pcap/elephant.pcap,infinite_rx=1 \
  -- -p config/patterns.txt -w 2 -B -M 3900000 2>&1 | tee logs/elephant_before.log
```
**Important**: with a multi-million-packet fixture, `infinite_rx=1` setup (caching the whole file into the mbuf pool) can take 30-100+ seconds before the process actually starts processing — watch for `[Worker 0] started on lcore` in the log before you start timing an observation window, and don't be surprised if `Ctrl+C`/`SIGTERM` takes a while to be honored if sent while it's still inside that setup call.

Once running, watch the `Per-worker PPS:` lines — the worker holding the elephant flow sustains massively higher PPS than the other(s) indefinitely with `-B` (a real run on this exact fixture showed a stable, unmoving `Bucket distribution: W0=128 W1=128` and PPS skew ranging roughly 1,000x-250,000x across ticks over 128 seconds, 0 rebalance events, as expected).

**After** (drop `-B`):
```sh
sudo stdbuf -oL -eL ./hyperdpi --lcores '(0-6)@(0-3)' -n 4 \
  --vdev=net_pcap0,rx_pcap=pcap/elephant.pcap,infinite_rx=1 \
  -- -p config/patterns.txt -w 2 -M 3900000 2>&1 | tee logs/elephant_after.log
```
Watch `Bucket distribution:` shift and `[Stats] rebalance: moved N bucket(s) ...` lines appear. A real 150-second run on this fixture produced **17 rebalance events** (8 buckets moved per pass, tapering to 5 then 1 as eligible idle buckets ran out) and `Bucket distribution` converged from `W0=128 W1=128` down to `W0=2 W1=254` — i.e. every bucket that safely could move, did. **Aggregate PPS on the overloaded worker did not drop much despite that** — this is expected, not a failure: the elephant flow alone is >99% of this fixture's packet volume, so redistributing the (individually tiny) mouse-flow buckets barely moves the aggregate number. The mechanism's actual job — keep new/idle traffic off the hot worker without ever touching the live elephant flow's own bucket — worked completely, as the bucket-distribution convergence shows directly.

### Verifying the rebalance mechanism directly (fast, no big fixture needed)

A standalone program that calls the bucket functions with controlled synthetic timestamps — no EAL/hugepages/root, no multi-GB fixture, no multi-minute run needed. Useful as a quick regression check after touching this code, complementing the full end-to-end run above:
```sh
cc -O0 -g -Iinclude -include rte_config.h $(pkg-config --cflags libdpdk libhs) \
   scripts/test_bucket_rebalance.c src/parser.c \
   $(pkg-config --libs libdpdk libhs) -o /tmp/test_bucket_rebalance
/tmp/test_bucket_rebalance
```
Expect `0 check(s) failed.` — covers even initial split, active buckets never moving, idle buckets moving correctly, the 8-per-pass anti-oscillation cap, and a regression check for the timestamp-underflow bug found during code review.

## Rule format (`config/patterns.txt`)

`<Traffic_Type>,<Regex_Pattern>,<Action>` — `Traffic_Type` is `HTTP` or `HTTPS`, `Action` is `DROP` or `FORWARD`. HTTPS patterns match against `HOST://<sni>` (note the `HOST://` prefix), HTTP patterns match against `http://<host><request-uri>`. No matching rule defaults to FORWARD.
